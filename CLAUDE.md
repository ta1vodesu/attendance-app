# Attendance App - 勤怠管理アプリ

---

## 1. 目的（誰が・何に困って・どう解決するか）

### 誰が困っているか
- 紙やExcelで勤怠を管理している中小企業の **総務・人事担当者**
- 打刻忘れや記録ミスが頻発し、月末に集計作業で残業が発生している
- 社員側も「自分の勤怠が正しく記録されているか」確認する手段がない

### 何に困っているか
- 打刻漏れ・修正依頼が口頭やメールで飛び交い、管理が属人化している
- 月末の集計作業に丸1日以上かかる
- 勤怠修正の承認フローが曖昧で、誰が承認したか記録が残らない
- リモートワーク増加で「出社していないと打刻できない」運用が破綻している

### どう解決するか
- Webブラウザからワンクリックで出退勤打刻できるシンプルなUIを提供
- 修正申請→承認の流れをシステム化し、証跡を残す
- 管理者向けダッシュボードで全社員の勤怠状況をリアルタイム把握
- 月次レポートをワンクリックで出力し、集計作業を自動化

## 2. 画面構成（2カラムレイアウト）

全画面共通で **左サイドバー + 右メインコンテンツ** の2カラム構成。
モバイルではサイドバーをハンバーガーメニューで折りたたむ。

### 共通画面（admin / member 両方）

| 画面名 | パス | 概要 |
|--------|------|------|
| ログイン | `/login` | メールアドレス+パスワードでログイン |
| ダッシュボード | `/dashboard` | 今日の打刻状況・お知らせ・直近の勤怠サマリ |
| 打刻 | `/attendance` | 出勤・退勤・休憩開始・休憩終了のボタン。現在のステータスを大きく表示 |
| 勤怠履歴 | `/history` | 自分の月別勤怠一覧。日付・出勤時刻・退勤時刻・勤務時間・ステータス |
| 修正申請 | `/history/:id/edit` | 特定日の勤怠に対して修正理由を添えて申請 |
| プロフィール | `/profile` | 自分の情報確認・パスワード変更 |

### 管理者専用画面（admin のみ）

| 画面名 | パス | 概要 |
|--------|------|------|
| 社員一覧 | `/admin/employees` | 全社員の一覧表示。検索・フィルタ・追加・編集・無効化 |
| 社員詳細 | `/admin/employees/:id` | 特定社員のプロフィール・勤怠サマリ |
| 全社勤怠一覧 | `/admin/attendance` | 全社員の日別勤怠を一覧表示。未打刻者のハイライト |
| 修正申請管理 | `/admin/approvals` | 未承認の修正申請を一覧表示。承認・却下ボタン付き |
| 部署管理 | `/admin/departments` | 部署の追加・編集・削除 |
| 月次レポート | `/admin/reports` | 月別の勤怠集計。CSV/PDF出力 |
| システム設定 | `/admin/settings` | 勤務時間ルール・休日設定・通知設定 |

### サイドバー構成

```
[ロゴ / アプリ名]
─────────────────
  ダッシュボード
  打刻
  勤怠履歴
─────────────────  ← admin のみ表示
  社員管理
  全社勤怠
  修正申請管理
  部署管理
  レポート
  設定
─────────────────
  [ユーザー名]
  [ログアウト]
```

## 3. 業務ルール（承認フロー・ステータス遷移）

### 打刻ステータス遷移

```
[未出勤] → 出勤打刻 → [勤務中] → 休憩開始 → [休憩中] → 休憩終了 → [勤務中] → 退勤打刻 → [退勤済]
```

- 1日に出勤・退勤は各1回のみ（重複打刻不可）
- 休憩は複数回取得可能（回数と合計時間を記録）
- 日付をまたぐ勤務（夜勤）は出勤日に紐づける
- 打刻忘れの場合、翌日以降に修正申請で対応

### 勤怠レコードのステータス

| ステータス | 意味 |
|-----------|------|
| `normal` | 正常に打刻完了 |
| `missing_clock_out` | 退勤打刻なし（打刻忘れ） |
| `correction_pending` | 修正申請中 |
| `correction_approved` | 修正承認済み |
| `correction_rejected` | 修正却下 |
| `absent` | 欠勤 |
| `paid_leave` | 有給休暇 |
| `holiday` | 休日 |

### 修正申請の承認フロー

```
[社員] 修正申請を提出
    ↓
ステータス: correction_pending
    ↓
[管理者] 申請一覧で確認
    ├── 承認 → ステータス: correction_approved → 勤怠レコード更新
    └── 却下 → ステータス: correction_rejected → 却下理由を記録
```

- 申請には「修正理由」が必須（例: 打刻忘れ、時刻誤り）
- 承認/却下時にタイムスタンプと承認者IDを記録（証跡）
- 却下された申請は再申請可能
- 管理者にはメールまたはアプリ内通知で新規申請を通知

### 勤務時間の計算ルール

- 勤務時間 = 退勤時刻 - 出勤時刻 - 合計休憩時間
- 所定労働時間: 8時間（システム設定で変更可能）
- 残業時間 = 勤務時間 - 所定労働時間（マイナスの場合は0）
- 深夜勤務（22:00-05:00）は別途カウント

## 4. 役割（管理者と一般ユーザーの違い）

### ロール定義

| ロール | 説明 |
|--------|------|
| `admin` | 管理者。全社員の勤怠管理・承認・設定変更が可能 |
| `member` | 一般社員。自分の打刻・履歴閲覧・修正申請のみ |

### 権限マトリクス

| 機能 | admin | member |
|------|:-----:|:------:|
| 出勤・退勤・休憩の打刻 | o | o |
| 自分の勤怠履歴閲覧 | o | o |
| 自分の勤怠に対する修正申請 | o | o |
| プロフィール編集 | o | o |
| **全社員の勤怠閲覧** | o | x |
| **修正申請の承認・却下** | o | x |
| **社員の追加・編集・無効化** | o | x |
| **部署の管理** | o | x |
| **月次レポート出力** | o | x |
| **勤務ルール・休日設定** | o | x |

### 画面の見え方の違い

- **サイドバー**: memberには管理者メニュー（社員管理〜設定）が表示されない
- **ダッシュボード**: adminには「未承認申請 X件」「本日未打刻者 X名」等の管理サマリが追加表示される
- **URLアクセス制御**: memberが `/admin/*` にアクセスした場合、ミドルウェアでダッシュボードにリダイレクト
- **API保護**: admin専用APIはミドルウェア+RLSの二重ガードで保護

## 5. 想定業務（どんな現場で・誰が・どう使うか）

### 想定ターゲット
- 従業員 10〜100名規模の中小企業
- 現状Excelや紙の出勤簿で勤怠管理している企業
- リモートワーク・フレックスタイム制を導入済み or 導入予定の企業

### 利用シーン

#### 一般社員の1日
1. **朝** — PCまたはスマホでログイン → 「出勤」ボタンを押す
2. **昼** — 「休憩開始」→ 昼食 → 「休憩終了」を押す
3. **夕方** — 業務終了時に「退勤」ボタンを押す
4. **月末** — 勤怠履歴で今月の勤務時間を確認。誤りがあれば修正申請

#### 管理者（総務担当）の1日
1. **朝** — ダッシュボードで昨日の未打刻者を確認 → 本人に連絡
2. **随時** — 修正申請の通知が届いたら内容を確認し、承認 or 却下
3. **月末** — 月次レポートを出力 → 給与計算システムに連携（CSV）
4. **随時** — 新入社員のアカウント作成、退職者の無効化

#### 管理者（経営層）
- 月次レポートで部署別の残業時間を把握
- 有給取得率を確認し、労務リスクを管理

## Architecture

### Tech Stack
- **Frontend**: Next.js (App Router) + TypeScript
- **UI**: Tailwind CSS + shadcn/ui
- **Backend**: Next.js API Routes (Route Handlers)
- **Database**: Supabase (PostgreSQL + Auth)
- **Hosting**: Vercel

### Database Schema

| テーブル | 用途 |
|---------|------|
| `profiles` | ユーザー情報（Supabase Auth連携、role, department_id） |
| `departments` | 部署マスタ |
| `attendance_records` | 勤怠レコード（clock_in, clock_out, breaks, status） |
| `attendance_corrections` | 修正申請（申請者, 対象日, 修正内容, 承認ステータス） |
| `system_settings` | 勤務ルール・休日などのシステム設定 |

### RLS (Row Level Security)
- memberは自身のレコードのみ SELECT/INSERT/UPDATE 可能
- adminは全レコードに対して CRUD 可能
- ミドルウェア + RLS の二重ガードでアクセス制御

## Conventions

### Coding
- 不変性（イミュータブル）を厳守
- 関数は50行以下、ファイルは800行以下
- Zodによる入力バリデーション必須
- エラーハンドリングは全APIルートで実施
- console.logは本番コードに残さない

### Git
- コミットメッセージ: `<type>: <description>`（feat, fix, refactor, docs, test, chore）
- ブランチ: `feature/`, `fix/`, `refactor/` プレフィックス

### Testing
- TDD（テスト駆動開発）を推奨
- カバレッジ目標: 80%以上
- E2Eテスト: 打刻フロー、ログイン、管理者操作を必須カバー
